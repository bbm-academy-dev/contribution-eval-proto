# contribution-eval-proto — Spec

**Дата:** 2026-04-27
**Статус:** дизайн утверждён, готов к реализации
**Контекст:** интерактивный одноразовый инструмент для проведения «Забега #1» — оценки вклада команды по итогам пилотного этапа OrthoBioSchool. Используется на одном звонке с командой.
**Методология:** [outputs/2026-04-27-zabeg1-contribution-evaluation-design.md](../outputs/2026-04-27-zabeg1-contribution-evaluation-design.md)

---

## Цели

- Провести оценку вклада 5–6 человек по двухслойной модели (продукты + glue) за один звонок.
- Дать команде живую визуализацию весов на этапе командной корректировки якоря.
- Автоматически считать доли и ₽ по формулам спеки.
- Запустить за один-два дня, задеплоить на Vercel, нулевая инфраструктура.

## Не-цели

- Многоразовое использование, шаблонизация под другие забеги.
- Сохранение состояния между сессиями (БД, аккаунты, авторизация).
- Защита от подмены имени, аудит, шифрование.
- Mobile-first дизайн (базовый responsive — да; но оптимизация под десктоп).

## Хардкод

3. `pilot_lesson` — Пилотный урок (тесты + презентация)

## Поток

### 1. Setup (фасилитатор, до или в начале звонка)

URL: `aggregate.html` → вкладка «Setup».

Вводит:
- **Список участников** — 5–6 имён, по одному на строку.
- **Якорная таблица** — для каждого из 4 продуктов: min ₽, max ₽. Mid-point = `(min+max)/2`. Стартовые веса вычисляются автоматически: `weight(P) = mid(P) / Σ mid(*)`. Σ mid-points = **якорный пул**.
- **Соотношение слоёв** — `product_ratio` (0–100%, дефолт 70). `glue_ratio = 1 − product_ratio`.

Жмёт «Сгенерировать ссылку для команды» → получает URL с параметрами в hash:
```
index.html#p=Anton,Eduard,Igor,Sasha,Masha&w=architecture:40,miniseries_ep1:25,pilot_lesson:15,content_factory:20&split=70
```
Кнопка «Копировать» → шарит ссылку в чат звонка.

### 2. Командная корректировка весов (на screen-share)

Фасилитатор остаётся на вкладке «Setup», шарит экран. Видны:

**A. Бар продукт/glue split** — горизонтальный стек 100% ширины с двумя сегментами и подписями (`Продукты 70% • Glue 30%`). Слайдер двигается, бар реактивно перерисовывается.

**B. Бары весов 4 продуктов** — четыре вертикальных бара, нормализованных по сумме = 100% продуктового слоя. У каждого:
- Подпись продукта сверху.
- Числовой инпут под названием — фасилитатор может вводить новое значение mid-point ₽ или сразу %, бары пересчитываются в реальном времени.
- Тонкий контур = якорный (стартовый) %.
- Заливка = текущий %.
- Дельта `40% → 38% (-5%)`. Если |дельта| > 20% — красная подсветка и предупреждение «выходит за коридор корректировки».
- Над баром: текущий % продуктового слоя + абсолют от пула (`28% от пула`).

Команда обсуждает корректировки вербально, фасилитатор вводит — все видят пересчёт. Финальные веса фиксируются (кнопка «Заморозить веса» → ссылка перегенерируется и шарится повторно, либо просто сохраняется в текущей сессии).

### 3. Заполнение формы участником (асинхронно, ~10–15 мин)

URL: `index.html#p=...&w=...&split=...`.

Шаги в форме:
1. **Выбор имени** — dropdown из переданного списка участников.
2. **Продуктовая секция** — для каждого из 4 продуктов:
   - Чекбокс «Я работал в этом продукте». Если не отмечен — секция продукта свёрнута, не учитывается.
   - Если отмечен:
     - Self: слайдер 0–100% «Какой % продукта сделал я».
     - Текстовое поле: «Что именно делал» (1–2 фразы).
     - Peer: 10 баллов forced-distribution между **остальными** участниками. На каждого — числовое поле + текстовое обоснование. Виджет показывает «Осталось распределить: N». Кнопка «Готово» по продукту неактивна, пока сумма ≠ 10.
3. **Glue секция** (всегда видна):
   - Текстовое поле «Что я внёс в общую ткань забега помимо продуктов» (обязательное, 1 абзац).
   - Peer: 10 баллов forced-distribution между **остальными** участниками (без самооценки). Аналогично — числовые поля + обоснования.
4. **Кнопка «Готово»** — активна, когда:
   - Имя выбрано.
   - В каждом отмеченном продукте сумма peer = 10.
   - Glue текст не пуст и сумма glue peer = 10.
   - Показывается текстовый блок:
     ```
     ZABEG-EVAL::eyJ2IjoxLCJuYW1lIjoiQW50b24i...
     ```
     (base64-encoded JSON, см. структуру ниже).
   - Кнопка «Скопировать в буфер».
   - Инструкция: «Вставьте этот блок в чат звонка».

### 4. Агрегация и расчёт (фасилитатор, на звонке)

URL: `aggregate.html` → вкладка «Свести результаты».

Вводит:
- **Блоки от участников** — одна большая textarea, парсер находит все строки начинающиеся с `ZABEG-EVAL::` (можно вставить всё сразу, разные форматы переноса строк ок).
- **Финальные веса** (после шага 2) — подтягиваются автоматически из setup, можно перепроверить.
- **Размер пула ₽** — числовое поле. Рядом отображается:
  - Якорный пул (Σ mid-points): `1 600 000 ₽`
  - Введённый пул: `1 800 000 ₽` → `+12.5%` ✓ в коридоре ±20%
  - Если пул выходит за ±20% — жёлтое предупреждение, не блок.

Жмёт «Посчитать» → вкладка автоматически переключается на «Результаты».

### 5. Результаты и калибровка

Вкладка «Результаты» показывает:

**A. Калибровочная таблица** — строки = участники, колонки = 4 продукта + glue. В каждой ячейке:
- Self % (для продуктов).
- Avg peer % (для продуктов и glue).
- Расхождение |self − avg_peer|. Если >30% — ячейка подсвечивается жёлтым.
- Кнопка-карандашик 📝 — открывает inline-редактор для **override**:
  - Можно править self или avg_peer (в исключительных случаях).
  - Поле «Заметка калибровки» (обязательное при правке).
  - Изменённая ячейка получает badge `📝 калибровка`, при наведении видна заметка.

Это и есть «один раз скорректировать peer» из шага 4 методологии — централизованно через фасилитатора по решению группы.

**B. Итоговая таблица** — на участника:
- Доля % (Σ всех источников)
- ₽ к выплате
- Stacked horizontal bar с разбивкой по 5 источникам (4 продукта + glue), длина = ₽, цвета сегментов соответствуют легенде.

**C. Сводки по слоям:**
- Продуктовый: `1 260 000 ₽` (70% × 1.8 млн)
- Glue: `540 000 ₽` (30% × 1.8 млн)
- Σ долей = 1.0, Σ выплат = пул (контроль).

**D. Кнопка «Экспорт»** — JSON и CSV (полная детализация: участник, источник, доля, ₽, заметки калибровки).

## Структура данных

### Evaluator-блок (то, что копирует участник)

JSON, base64-encoded, префикс `ZABEG-EVAL::`:

```json
{
  "v": 1,
  "name": "Anton",
  "ts": "2026-04-27T12:34:56Z",
  "products": {
    "architecture": {
      "self": 60,
      "what": "разработал слой API + схему данных",
      "peer": { "Eduard": 0, "Igor": 2, "Sasha": 5, "Masha": 3 },
      "peer_notes": { "Igor": "ревью схемы", "Sasha": "сделал основной фронт", "Masha": "тестирование" }
    },
    "miniseries_ep1": null,
    "pilot_lesson": null,
    "content_factory": {
      "self": 40,
      "what": "архитектура AI-конвейера",
      "peer": { "Eduard": 1, "Igor": 0, "Sasha": 6, "Masha": 3 },
      "peer_notes": { "Eduard": "конкретика по фарма-требованиям", "Sasha": "сценарная часть", "Masha": "monitoring" }
    }
  },
  "glue": {
    "what": "координировал команду, держал общий темп, делал стратегическую сборку",
    "peer": { "Eduard": 4, "Igor": 3, "Sasha": 2, "Masha": 1 },
    "peer_notes": { "Eduard": "фарма-доступы", "Igor": "стратегия и инвест-связь", "Sasha": "ресёрч", "Masha": "поддержка" }
  }
}
```

### Setup-параметры в URL (hash, не query — чтобы не попадало на сервер при ошибке)

```
index.html#p=Anton,Eduard,Igor,Sasha,Masha&w=architecture:40,miniseries_ep1:25,pilot_lesson:15,content_factory:20&split=70
```

- `p` — comma-separated имена (URL-encoded).
- `w` — `productKey:percent` пары через запятую. Сумма = 100.
- `split` — product_ratio в процентах (0–100).

## Алгоритм расчёта (`shared.js`)

Pseudo-code:

```
function compute(blocks, finalWeights, productRatio, pool):
    glueRatio = 1 - productRatio
    participants = unique(blocks.map(b => b.name))

    productShare = {}
    for P in products:
        contributors = blocks.filter(b => b.products[P] != null)
        if contributors.empty: continue

        // self-доли по продукту
        selfSum = Σ contributor.products[P].self
        selfShare = { c.name: c.products[P].self / selfSum for c in contributors }

        // peer-доли по продукту
        peerTotal = 0
        peerByPerson = {}
        for c in contributors:
            for (target, points) in c.products[P].peer:
                peerByPerson[target] += points
                peerTotal += points
        peerShare = { name: pts / peerTotal for (name, pts) in peerByPerson }

        for X in participants:
            shareInP = 0.5 * (selfShare[X] || 0) + 0.5 * (peerShare[X] || 0)
            productShare[X] += finalWeights[P] * shareInP

    // glue
    glueTotal = 0
    glueByPerson = {}
    for b in blocks:
        for (target, points) in b.glue.peer:
            glueByPerson[target] += points
            glueTotal += points
    glueShare = { name: pts / glueTotal for (name, pts) in glueByPerson }

    // итог
    finalShare = {}
    pay = {}
    for X in participants:
        finalShare[X] = productRatio * (productShare[X] || 0)
                      + glueRatio    * (glueShare[X]    || 0)
        pay[X] = pool * finalShare[X]

    assert Σ finalShare ≈ 1.0
    assert Σ pay ≈ pool
    return { productShare, glueShare, finalShare, pay }
```

Override (калибровка): после расчёта фасилитатор может править `selfShare[X][P]` или `peerShare[X][P]` напрямую — пересчёт `productShare[X]` и далее по цепочке. Override-значения сохраняются в state с заметкой.

## UI / визуализация

**Без библиотек.** SVG для баров и stacked-bars, CSS-flex для layout. Нативные `<input type="range">`, `<input type="number">`, `<details>`.

**Цветовая палитра** (минимум):
- Продукты: 4 разных оттенка (например, синий/голубой спектр).
- Glue: контрастный (оранжевый/янтарный).
- Подсветки: жёлтая (>30% расхождение, выход за ±20%), красная (выход за коридор веса).

**Шрифты:** системные (system-ui sans-serif).

**Responsive:** базовый — стек на узких экранах, без специальной мобильной оптимизации.

## Тестирование

Минимум — ручной прогон по сценарию:
1. Setup с тестовыми данными (3 фейковых участника, фейковый якорь).
2. Открыть форму, заполнить от 3 разных «лиц», получить блоки.
3. Вставить в агрегатор, проверить:
   - Σ долей = 1.0 (с точностью до 0.001).
   - Σ выплат = пул (с точностью до 1₽).
   - Override работает, заметка сохраняется.
   - Подсветка >30% и ±20% работают.

Юнит-тестов не пишем (одноразовый прототип); если в `shared.js` не сходятся суммы — отлаживаем по browser console.
